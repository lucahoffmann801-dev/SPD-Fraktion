## What this project is

Internal portal for the SPD-Fraktion Kaiserslautern (German city council faction).  
Features: calendar/events, tasks, political cases, committee memberships, documents, member profiles, ICS calendar sync.

**Language of the UI**: German. All labels, button text, section titles, and user-facing strings are in German. Keep this when editing UI code.

All CSS lives in `artifacts/portal/src/`:

- `globals.css` — main stylesheet, mobile breakpoint block at bottom

- `work-orders.css` — work order / task styles
- `termine-filter.css` — event filter styles
- `progress-slider.css` — progress bar / slider styles
- `task-assignment.css` — assignee picker styles

**Mobile breakpoint**: `@media (max-width: 860px)` — all mobile overrides go inside this block.  
**iOS safe areas**: use `env(safe-area-inset-bottom)` for tab bar and bottom sheets.  
**No Tailwind** — plain CSS custom properties only. Variables defined in `globals.css` (`:root { --red: ...; --surface: ...; ... }`).

## Architecture decisions (critical — do not violate)

### 1. Single-page app, no router
`home.tsx` is the entire portal UI. Navigation is a React `useState` called `view`. There is no React Router, no Wouter routing in the portal. Do not add page routing.

### 2. Demo-mode fallback (always present)
The app works fully without Supabase. Every API route checks `if (!supabase) return res.json(demoData)`. Do not remove this pattern. Demo data is embedded inline in the route files (not imported from separate files).

### 3. Supabase key separation
- `VITE_SUPABASE_ANON_KEY` — browser client, read-only, respects Row Level Security
- `SUPABASE_SERVICE_ROLE_KEY` — server only, full access, bypasses RLS
- Never expose the service role key to the browser.

### 4. Generated files — never edit by hand
`lib/api-zod/src/` and `lib/api-client-react/src/` are auto-generated by Orval from the OpenAPI spec.  
To change: edit `lib/api-spec/openapi.yaml`, then run `pnpm --filter @workspace/api-spec run codegen`.

### 5. Description-field markers (tasks)
The `tasks.description` field encodes metadata as inline text markers (legacy Supabase compatibility):
- `[progress:75]` — progress percentage
- `[assignees:slug1,slug2]` — multiple assignees
- `[visible:private:slug1,slug2]` — restrict visibility
- `[Patrick→Luca]` or `[Patrick-Luca]` — marks as a work order

The API server (`portal-records.ts` `normalizeTaskPayload()`) strips/parses these on every write. Do not bypass this normalization.

### 6. Upload authentication
Upload tokens are HMAC-signed JWTs issued at login for permitted profiles.  
Upload permission slugs are hardcoded in `artifacts/api-server/src/lib/uploadAuth.ts`.  
Token is sent as `X-Portal-Upload-Token` header to `POST /api/storage/uploads/request-url`.

## Database schema (Supabase / PostgreSQL)

Tables live in Supabase. Schema types are in `lib/db/src/schema/`.

### `profiles`
Portal login accounts.  
Key fields: `id` (e.g. `p-01`), `slug` (e.g. `patrick-schaefer`), `full_name`, `display_name`, `role`, `portal_role`, `permissions` (array of strings), `accent` (color name), `sort_order`, `login_enabled`.

`portal_role` values: `fraktionsvorsitz`, `stellvertretung`, `ratsmitglied`, `fraktionssekretariat`  
`permissions` values: `termine`, `aufgaben`, `dokumente`, `kalender`, `profile`, `admin`, `admin-lite`

### `events`
Calendar events. Key fields: `id`, `title`, `starts_at`, `ends_at`, `all_day`, `location`, `category`, `source`, `status`, `meeting_body`, `preparation_status`, `requires_preparation`, `decision_needed`.

### `tasks`
Tasks/work orders. Key fields: `id`, `title`, `description` (with inline markers), `assignee`, `assignees` (array), `status` (`offen`/`in-bearbeitung`/`erledigt`/`verworfen`), `priority`, `progress` (0–100), `is_work_order`, `visibility`, `visible_to` (array).

### `members`
Public council member list. Fields: `id`, `name`, `role`, `committees` (array).

### `documents`
Uploaded files. Fields: `id`, `title`, `storage_path`, `content_type`, `size`, `uploaded_by`, `created_at`, `category`.

### `committees` / `committee_memberships`
`committees`: `id`, `name`, `short_name`, `category`  
`committee_memberships`: `id`, `profile_id`, `committee_id`, `role` (`mitglied`/`stellvertreter`)

### `cases`
Political cases/issues. Fields: `id`, `title`, `description`, `status`, `category`, `assignee`.

### `calendar_sources`
ICS sync sources. Fields: `id`, `name`, `url`, `active`, `last_synced_at`.

## Common tasks

### Add a new view/section to the portal

1. Add a new tab to the bottom nav in `home.tsx` (mobile) and sidebar (desktop)
2. Add a case to the `view` state switch in the render logic
3. Add the view component inline in `home.tsx` (keep everything in one file)
4. Add any new CSS class to the appropriate CSS file in `artifacts/portal/src/`

### Add a new API route

1. Create `artifacts/api-server/src/routes/portal-<name>.ts`
2. Register in `artifacts/api-server/src/routes/index.ts`
3. If it's a new endpoint shape: add to `lib/api-spec/openapi.yaml` and run codegen

### Add a new database table

1. Define in `lib/db/src/schema/index.ts` using Drizzle's `pgTable`
2. Export the table, insert schema, and types
3. Run `pnpm --filter @workspace/db run push` (dev) or `generate` + `migrate` (production)
4. Add the table name to `ALLOWED_TABLES` in `portal-records.ts` if it needs CRUD

### Change profile images

Profile images live in `artifacts/portal/public/profile-images/`.  
The mapping `profileImageBySlug` in `home.tsx` maps slugs to image paths.  
Format: JPEG, named `<slug>.jpeg`.

### Regenerate API types after spec change

```bash
pnpm --filter @workspace/api-spec run codegen
```

### Push schema to database

```bash
pnpm --filter @workspace/db run push
```

## Gotchas

- **`home.tsx` is intentionally large** — all views are inline. Do not refactor into separate files unless explicitly asked.
- **CSS mobile block**: Mobile overrides must be inside `@media (max-width: 860px)` in `globals.css`. Accidentally placing rules outside will affect desktop.
- **pnpm only** — `npm install` or `yarn` will be rejected by the preinstall hook.
- **Generated files**: Never edit `lib/api-zod/src/` or `lib/api-client-react/src/` manually. Always use codegen.
- **Supabase null check**: `getSupabase()` can return `null`. Every route must handle this gracefully with a demo-data fallback.
- **Task description markers**: The description field is a string that encodes structured data. Always use `normalizeTaskPayload()` when writing tasks.
- **esbuild version**: Pinned to `0.27.3` in the workspace (overrides) — do not bump without checking compatibility.
- **React version**: Pinned to `19.1.0` (exact) because Expo requires it. Do not change.
